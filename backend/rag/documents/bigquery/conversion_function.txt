
BIGQUERY CONVERSION FUNCTIONS 
This document provides a narrative, context-heavy overview of GoogleSQL for BigQuery’s conversion functions. It follows the same semantic density and structural format as prior documents, ensuring it is well suited for Retrieval-Augmented Generation (RAG) pipelines. Each section is self-contained and crafted for high‑quality embeddings.

---
SECTION: OVERVIEW OF CONVERSION FUNCTIONS

BigQuery’s conversion functions transform values between SQL types, JSON types, arrays, bytes, numbers, dates, times, timestamps, and encoded text formats. These conversions may be explicit—performed intentionally using CAST or specialized parsing functions—or implicit, occurring when BigQuery coerces values to meet function signature requirements. Conversion functions allow data normalization, interoperability between structured and semi‑structured formats, human-readable outputs, encoded byte transformations, and safe handling of uncertain inputs. Together, they form the foundational translation layer between heterogeneous data representations.

---
SECTION: ARRAY_TO_STRING – ARRAY TO TEXT CONCATENATION

ARRAY_TO_STRING transforms an array of values into a single STRING by joining its elements with a delimiter. This is essential for presentation, reporting, logging, and storing array contents in text fields. It also supports optional null-handling logic, enabling substitution of placeholders for NULL array entries. Although simple, it plays a key role in transitioning from structured array data to flat textual output.

---
SECTION: CAST – EXPLICIT DATA TYPE CONVERSION

CAST performs explicit conversion of an expression into a target type. It is strict: if conversion is impossible—such as interpreting invalid UTF‑8 bytes as a STRING—an error occurs. CAST may include a format clause, which provides nuanced control over how values are interpreted, particularly when converting BYTES or parsing text into temporal types. CAST is used for schema alignment, interoperability with external data formats, and consistent type enforcement in analytic logic. SAFE_CAST is the safer variant that returns NULL instead of failing, preventing runtime errors in production pipelines.

---
SECTION: CHR AND UNICODE CODE POINT FUNCTIONS

CHR converts a Unicode code point into the corresponding character, enabling construction of strings from numeric representations. CODE_POINTS_TO_STRING and CODE_POINTS_TO_BYTES perform the inverse operations, assembling text or byte sequences from arrays of numeric code points. These functions are fundamental in low-level encoding tasks, character-level transformation, and programmatic string construction in ETL workflows.

---
SECTION: BASE ENCODING AND HEXADECIMAL TRANSFORMATION

BigQuery supports several encoding and decoding functions: FROM_BASE32, FROM_BASE64, FROM_HEX, TO_BASE32, TO_BASE64, and TO_HEX. These functions translate between raw BYTES values and textual encodings used for transport, storage, and interoperability. They enable reversible transformations suitable for cryptographic operations, compact data serialization, and compatibility with external APIs and file formats. SAFE_CONVERT_BYTES_TO_STRING complements these by replacing invalid byte sequences with the Unicode replacement character, ensuring robust conversions even with malformed inputs.

---
SECTION: JSON CONVERSION FUNCTIONS

JSON-centric conversion functions bridge semi‑structured JSON content and BigQuery’s strongly typed SQL system. Functions like BOOL(JSON), INT64(JSON), FLOAT64(JSON), STRING(JSON), and their LAX_ variants translate JSON primitives into SQL values. LAX_ functions return NULL instead of failing, supporting tolerant ingestion of inconsistent or messy JSON payloads. PARSE_JSON converts a JSON-formatted STRING into a JSON typed value, and TO_JSON / TO_JSON_STRING serialize SQL values into canonical JSON structures. These tools power flexible handling of modern, nested, schema-light data.

---
SECTION: TEMPORAL PARSING AND CONVERSION FUNCTIONS

PARSE_DATE, PARSE_DATETIME, PARSE_TIME, and PARSE_TIMESTAMP translate textual representations of date and time into canonical BigQuery temporal types. Each relies on customizable format strings. TIMESTAMP_MICROS, TIMESTAMP_MILLIS, and TIMESTAMP_SECONDS convert epoch-based counters into absolute instants. Conversely, UNIX_DATE, UNIX_MICROS, UNIX_MILLIS, and UNIX_SECONDS convert temporal values back to numeric counters suitable for efficient storage and sorting. These functions provide deterministic mappings between human-readable and machine-oriented time formats.

---
SECTION: NUMERIC PARSING FUNCTIONS – PARSE_NUMERIC AND PARSE_BIGNUMERIC

PARSE_NUMERIC and PARSE_BIGNUMERIC interpret STRING literals as NUMERIC or BIGNUMERIC values. They enforce strict precision rules, rejecting strings exceeding allowed ranges or fractional limits. These functions enable ingestion of financial, scientific, or high-precision numerical data originating from text-based formats such as CSV files.

---
SECTION: CASTING TO SPECIFIC SQL TYPES

BigQuery defines precise conversion rules when casting to particular SQL types:

• CAST AS BOOL interprets INT64 zero as FALSE, non-zero as TRUE, and STRING values “true” or “false” (case-insensitive). Other strings produce errors.
• CAST AS INT64 converts numeric strings or floating-point values when possible, but rejects invalid textual forms.
• CAST AS FLOAT64 accepts INT64, NUMERIC, and BIGNUMERIC by approximation, and supports INF and NAN textual forms.
• CAST AS DATE, DATETIME, TIME, and TIMESTAMP interpret STRINGS using format clauses or literal rules, applying truncation rules when converting from TIMESTAMP.
• CAST AS BYTES encodes strings using UTF‑8 unless a format model specifies hexadecimal or BASE encoding behavior.
• CAST AS ARRAY requires exact element type compatibility.
• CAST AS BIGNUMERIC or NUMERIC enforces precision and rounding rules, rejecting out-of-range values.

These casts ensure predictable transformation across domains while preserving data validity.

---
SECTION: SAFE_CAST – FAULT-TOLERANT CONVERSION

SAFE_CAST provides the same functionality as CAST but returns NULL instead of raising errors. It is widely used in large-scale ingestion pipelines where occasional malformed data must not terminate execution. SAFE_CAST supports robust, defensive programming by insulating business logic from upstream data inconsistencies.

---
SECTION: UNIX AND EPOCH CONVERSIONS

UNIX_DATE, UNIX_MICROS, UNIX_MILLIS, and UNIX_SECONDS convert DATE or TIMESTAMP values into epoch-based numeric counters. These are widely used in partitioning, range joins, temporal bucketing, machine learning feature creation, and low-level storage structures. They complement the reverse converters TIMESTAMP_MICROS, TIMESTAMP_MILLIS, TIMESTAMP_SECONDS, and DATE_FROM_UNIX_DATE.

---
SECTION: ARRAY, STRING, AND BYTES TRANSFORMERS

TO_CODE_POINTS converts STRING or BYTES into arrays of integers representing ASCII code points, enabling detailed inspection and transformation. ARRAY_TO_STRING reverses array contents into flattened strings. These utilities support flexible structural manipulation, making them essential in dynamic scripting, text mining, and API integration.

SECTION: CAST AS INT64 – INTERPRETING VALUES AS INTEGER DOMAIN

Casting to INT64 transforms an input expression into a 64‑bit signed integer. BigQuery supports this conversion from floats, numerics, booleans, and specific string literals. FLOAT64 conversions round to the nearest integer, with midpoint values rounding away from zero—ensuring deterministic handling of edges such as 1.5 becoming 2, or -0.5 becoming -1. BOOL values map to 1 for TRUE and 0 for FALSE, enabling logical flags to integrate with arithmetic workflows. STRING inputs accept ordinary integer literals as well as hexadecimal forms prefixed by 0x or -0x, translating them into their integer equivalents. This capability supports flexible ingestion of numeric text, hex-encoded identifiers, and user‑supplied input.

---
SECTION: CAST AS INTERVAL – INTERPRETING DURATION STRINGS

Casting to INTERVAL interprets a STRING as a civil duration. Accepted forms include the ISO 8601 duration standard—using P‑prefixed notation—and BigQuery’s interval literal format expressed as ‘Y-M D H:M:S.F’. Partial forms may be used when they remain unambiguous, such as ‘H:M:S’ or ‘1-2’ for years‑months. Casting enforces structural validity and range boundaries: malformed strings or durations beyond supported intervals produce runtime errors. These conversions enable ingestion of human-entered durations, configuration metadata, and external system interval encodings directly into BigQuery’s temporal arithmetic engine.

---
SECTION: CAST AS NUMERIC – PRECISION‑SAFE DECIMAL CONVERSION

Casting to NUMERIC converts INT64, FLOAT64, NUMERIC, BIGNUMERIC, or STRING expressions into a fixed‑precision decimal with 38 digits of precision and 9 digits of scale. FLOAT64 inputs undergo half‑away‑from‑zero rounding, rejecting NaN and infinity. STRING inputs must represent valid decimal numbers within NUMERIC’s domain, respecting the nine‑digit scale limit; excessive fractional digits are rounded to scale. Out‑of‑range inputs throw overflow errors. This casting mechanism is important for financial systems, accounting transformations, and high‑integrity numerical pipelines.

---
SECTION: CAST AS RANGE – STRUCTURED INTERVAL BOUNDARIES

Casting to RANGE interprets a STRING literal into a RANGE over DATE, DATETIME, or TIMESTAMP. The string must follow canonical range syntax such as ‘[lower, upper)’ or use UNBOUNDED/NULL markers. Conversions validate subtype compatibility, boundary formatting, and domain limits. Incorrect lower/upper formats or invalid time-zone components generate errors. Range casting is central in analytic use cases like period bucketing, event segmentation, time-window modeling, and bi-temporal logic.

---
SECTION: CAST AS STRING – HUMAN-READABLE SERIALIZATION

Casting to STRING provides a textual representation of SQL values. Supported inputs include numeric, boolean, temporal, bytes, range, and interval types. Temporal conversions follow canonical structures: TIME becomes HH:MM:SS, DATE becomes YYYY-MM-DD, DATETIME becomes YYYY-MM-DD HH:MM:SS, and TIMESTAMP is rendered in UTC unless overridden with a format clause or AT TIME ZONE directive. BYTES input interprets raw bytes as UTF‑8 unless a format clause specifies alternative encodings such as HEX, BASE32, BASE64, or ASCII. Numeric conversions support formatted layouts via explicit format patterns. INTERVAL casting emits a structured representation ‘Y-M D H:M:S.F’. These conversions underpin reporting pipelines, audit logs, UI formatting, serialization, and exports.

---
SECTION: CAST AS STRUCT – HIERARCHICAL TYPE ALIGNMENT

Casting to STRUCT requires structural compatibility: both the source and target structs must have the same number of fields, and each field must be explicitly castable to the corresponding target field’s type, following positional—not named—ordering. This supports controlled schema transitions, backward‑compatible transformations, and typed struct evolution during ETL workflows.

---
SECTION: CAST AS TIME – PARSING AND NORMALIZING CIVIL TIMES

Casting to TIME accepts STRING, TIME, DATETIME, and TIMESTAMP. STRING inputs must follow supported time literal formats, including fractional seconds, while remaining independent of time zones. Invalid or out‑of‑range expressions fail conversion. Casting from DATETIME or TIMESTAMP extracts only the civil time component; TIMESTAMP conversions interpret instants relative to UTC or to a specified time zone via format clauses. This enables consistent normalization of timestamped events into pure time-of-day semantics.

---
SECTION: CAST AS TIMESTAMP – ABSOLUTE INSTANT CONVERSION

Casting to TIMESTAMP transforms STRING, DATE, or DATETIME into an absolute point on the UTC timeline. STRING inputs must conform to supported timestamp literal formats and may include embedded time zones. If no timezone is embedded, UTC or an explicitly declared AT TIME ZONE clause defines interpretation. Datetime-to-timestamp conversions apply civil-to-absolute mapping rules, accounting for daylight savings transitions: ambiguous times select the earlier corresponding timestamp, and nonexistent times advance through the DST gap. Overflow, invalid literals, or excessive subsecond precision produce errors. This conversion is foundational for event-time analytics, ordering, windowing, and cross‑system synchronization.

---
SECTION: PARSE_BIGNUMERIC – HIGH‑PRECISION TEXTUAL DECIMAL INTERPRETATION

PARSE_BIGNUMERIC reads a STRING as a high‑precision decimal with approximately 76 digits of precision. It accepts flexible formatting including whitespace, exponent notation, trailing signs, commas in the integer portion, and leading-dotted decimals. Strict rules prohibit multiple signs, invalid characters, misplaced commas, or digit interruptions. Excess fractional digits are rounded half‑away‑from‑zero to 38 decimal places. Values exceeding BIGNUMERIC’s range generate errors. This is critical for scientific, economic, and mathematical ingestion workflows where precision surpasses NUMERIC’s limits.

---
SECTION: PARSE_NUMERIC – INGESTING FIXED‑PRECISION NUMBERS

PARSE_NUMERIC functions similarly to PARSE_BIGNUMERIC but targets NUMERIC’s 38-digit precision and 9-digit scale. It supports flexible sign placement, comma grouping, exponent syntax, and whitespace tolerance. Digit gaps, invalid characters, multiple signs, or out-of-range magnitudes trigger errors. Excess fractional digits undergo rounding. This function enables ingestion of human-entered decimal values while enforcing NUMERIC constraints.

---
SECTION: SAFE_CAST – NON-FAILING TYPE CONVERSION

SAFE_CAST mirrors CAST but replaces runtime failures with NULL, preventing query failures caused by malformed values. Static analysis still enforces impossible casts—for example BOOLEAN to STRUCT. SAFE_CAST is especially valuable in ETL pipelines ingesting heterogeneous or messy data: instead of terminating execution, it yields NULL, allowing post-processing logic to handle exceptions gracefully. SAFE_CONVERT_BYTES_TO_STRING provides similar resilience when interpreting byte sequences, replacing invalid UTF‑8 characters with the Unicode replacement character.
